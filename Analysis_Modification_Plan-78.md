# Analysis and Modification Plan - 78

## 1. 核心架构选择：分布式 vs 集中式

针对您提出的存储位置问题，为了平衡**数据安全性（容灾）**与**检索性能（速度）**，新方案建议采用 **“分布式物理存储 + 集中式运行时快照”** 的混合二进制架构，而非简单地二选一。

### 1.1 事实来源 (Source of Truth)：分布式存储
**方案：依然在每个文件夹下生成一个 `metadata.scch`（二进制版）。**
- **原因 (为什么不放在一个文件里？)**:
    - **移动性**: 当您移动、重命名或备份某个文件夹时，其元数据（标签、颜色、评分）会随之移动。如果全部记录在一个中心文件里，一旦路径发生变化，所有元数据都会丢失。
    - **安全性**: 避免单点故障。如果一个总的二进制文件损坏，您将丢失全盘所有记录；分布式存储确保即便某个文件夹损坏，其他数据依然安全。
    - **去数据库化**: 集中式大文件本质上就是另一种形式的“数据库”，违背了彻底废除 DB 逻辑的初衷。

### 1.2 性能引擎 (Performance Engine)：集中式二进制快照
**方案：在程序启动时或扫描后，于内存/临时目录生成一个全局的二进制索引快照（Global Snapshot）。**
- **作用**: 解决搜索性能问题。当您搜索某个标签时，程序不需要去遍历全盘数万个文件夹，而是直接查询这个极轻量、纯二进制的内存索引。
- **与 DB 的区别**:
    - 它是**不可变/临时**的。它只是分布式数据的映射，随时可以删除并重新生成。
    - 它不使用复杂的 SQL 引擎，而是使用简单的偏移量查找，速度极快。

## 2. 存储协议的物理实现

在 **Plan-76** 定义的二进制协议基础上，每个文件夹内的 `.scch` 文件将包含：
1. **本文件夹元数据** (FolderMeta)。
2. **本文件夹内所有文件的元数据列表** (ItemMeta List)。

这确保了每个文件夹都是一个自包含的元数据节点。

## 3. 逻辑流程演进

1. **写入**: 当您在 UI 修改一个文件的颜色时，`MetadataManager` 直接定位到该文件所在的目录，重写该目录下的二进制 `.scch` 文件。
2. **加载**: 启动时，系统通过 `MftReader` 极速定位全盘所有的 `metadata.scch` 文件，并将关键索引信息读入内存快照。
3. **查询**: 搜索操作完全在内存快照中进行，不产生磁盘 IO。

## 4. 结论
新方案**依然保留“每个文件夹一个 SCCH 文件”的设计**，以确保元数据的物理便携性与安全性；但通过**纯二进制格式**（取代旧版 JSON）和**全局内存映射**解决了性能问题。这既不是旧版的低效 JSON，也不是笨重的 SQL 数据库，而是现代化的、工业级的离散元数据管理方案。
