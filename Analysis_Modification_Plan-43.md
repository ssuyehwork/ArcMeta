**1.2 现有架构问题定位**
- 问题一：**读写锁竞争导致的“写者饥饿”与 UI 挂起**
  - 根因：`saveToCache()` 在执行 O(N) 遍历及磁盘 I/O 期间始终持有 `m_dataLock` 的 **读锁 (QReadLocker)**。虽然读锁允许多个读者并发，但一旦有写者（如 `UsnWatcher` 线程尝试通过 `updateEntryFromUsn` 获取写锁）进入等待队列，后续的所有读者请求（包括 UI 线程的搜索、刷新、渲染）都会被阻塞在写者之后。
  - 结果：磁盘 I/O 的 1-2 秒耗时变成了 UI 线程的 1-2 秒僵死，导致 MainWindow 失去响应。

- 问题二：**UI 线程执行同步忙等 (Busy-wait)**
  - 根因：在 `MftReader::clear()` 中，为了安全释放内存，代码使用了 `while(m_is_saving) { msleep(10); }`。由于 `ScanDialog` 的析构函数在 UI 线程运行，如果后台存盘任务正在进行，UI 线程将在此处循环阻塞。

- 问题三：**逻辑可见性缺失**
  - 根因：`getPathFast` 及部分 SoA 访问接口未加锁。虽然目前未崩溃，但在高并发变动下存在读取非法内存的隐患。

### § 2 方案设计（Solution Architecture）

**2.1 技术选型决策**
| 候选方案 | 优点 | 缺点 | 是否采用 | 理由 |
|----------|------|------|----------|------|
| 方案 A：持有锁执行 I/O | 实现极其简单 | 严重阻塞，UI 频繁假死 | ❌ 放弃 | 典型的“小学生”并发 Bug |
| 方案 B：锁内采样，锁外 I/O | 锁持有时间极短 | 增加内存拷贝开销 | ✅ 采用 | 工业级高性能并发标准做法 |
| 方案 C：clear() 异步化 | 毫秒级关窗响应 | 需处理生命周期边界 | ✅ 采用 | 彻底解决关窗卡顿 |

**2.2 目标架构设计**
- 核心改进：
  1. **锁分离技术**：重构 `saveDriveToCacheInternal`。仅在内存遍历和数据“采样”（Snapshotting）阶段持有读锁。将耗时的 `ScchCache::save` (磁盘写入) 移出锁保护范围。
  2. **异步清理任务**：`MftReader::clear()` 改为“非阻塞”模式。立即标记 `m_isInitialized = false` 使 UI 失效，然后启动后台任务执行剩余的等待、存盘及内存物理释放操作。
  3. **读写优先级优化**：通过缩短读锁持有时间，消除写者排队导致的 UI 读者挂起。

### § 3 性能影响评估
- **UI 响应性**：UI 线程被锁阻塞的概率从 O(IO_Time) 降至 O(Memory_Copy_Time)，约提升 1000x。
- **内存回收**：保持物理归还机制，且不再阻塞窗口关闭。
