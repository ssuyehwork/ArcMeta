**1.2 现有架构问题定位**
- 问题一：`MftReader::updateEntryFromUsn` 中的致命性能阻塞（约 839 行）
  - 根因：在持有 `m_dataLock` 的 **写锁（QWriteLocker）** 期间调用了 `saveDriveToCacheInternal(dIdx)`。`saveDriveToCacheInternal` 是一个 O(N) 操作（N 为全盘文件总数），涉及遍历数百万条目、字符串拷贝及磁盘 I/O。
  - 影响面：由于写锁是排他的，这会导致 UI 线程（搜索、渲染）和所有其他工作线程完全阻塞，直到 I/O 完成。每 1000 次 USN 变动触发一次，在高频率文件操作下程序必然假死。

- 问题二：`MftReader::clear()` 在 UI 线程同步执行耗时 I/O（约 106 行）
  - 根因：`ScanDialog` 析构时调用 `clear()`，而 `clear()` 现在同步执行 `saveToCache()`。对于大容量磁盘，这可能导致 UI 线程阻塞数秒甚至数十秒，触发 Windows 的“程序无响应”对话框。

- 问题三：`saveDriveToCacheInternal` 效率低下（约 365 行）
  - 根因：每次保存单个盘符的缓存时，都执行 `for (size_t i = 0; i < m_frns.size(); ++i)` 全量遍历。如果挂载了 5 个盘，每个盘更新都会扫描全量内存池，浪费严重。

**1.3 调用链 / 数据流图**
```
UsnWatcher 线程
  └─► handleRecord()
        └─► MftReader::updateEntryFromUsn()
              └─► lock(m_dataLock.Write)  <-- 锁定全场
                    └─► saveDriveToCacheInternal() <-- 耗时 I/O (1s+)
                          └─► UI 线程 (搜索/渲染) 被迫挂起等待锁
```

### § 2 方案设计（Solution Architecture）

**2.1 技术选型决策**
| 候选方案 | 优点 | 缺点 | 是否采用 | 放弃原因 |
|----------|------|------|----------|----------|
| 方案 A：锁内 I/O | 实现简单 | 严重卡顿，UI 假死 | ❌ 放弃 | 性能不可接受 |
| 方案 B：异步 I/O 线程 | 不阻塞主流程 | 需处理并发数据快照一致性 | ✅ 采用 | 工业级标准做法 |
| 方案 C：延迟批量持久化 | 减少 I/O 频率 | 掉电丢失风险稍增 | ✅ 采用 | 平衡性能与可靠性 |

**2.2 目标架构设计**
- 设计原则遵循：**非阻塞 I/O** 与 **读写分离**。
- 核心改动逻辑：
  1. 将 `updateEntryFromUsn` 中的 `saveDriveToCacheInternal` 移出写锁范围，或者改为发送信号/推入任务队列。
  2. `clear()` 中的 `saveToCache()` 逻辑：如果是为了持久化 USN 游标，可以仅快速持久化游标，或者在关闭前确保异步持久化已完成。
  3. 优化 `saveDriveToCacheInternal`：仅处理变更盘符，避免全量 O(N) 扫描（通过预过滤或按盘符隔离的存储结构，但当前暂不重构 SoA 结构以保持稳定，先解决锁阻塞）。
