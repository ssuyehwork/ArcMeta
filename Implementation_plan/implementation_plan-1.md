# 修复托管库自动入库架构缺陷与视觉欺骗逻辑 —— implementation_plan-1.md

## 1. 任务背景
用户反馈在拖拽项目（尤其是同盘符移动）到托管文件夹后，系统显示“完全且完美登记注册”（对应用户原话），但实际未完成登记。经审计发现系统在 USN 信号处理、元数据状态机、路径缓存清理及 UI 默认值设定上存在“傻逼逻辑架构”（对应用户原话）。

## 2. 问题定位
- **入库信号缺失**：`AutoImportManager::onEntryUpdated` 仅处理重命名，缺失了新项移入库的准入校验，导致同盘移动操作无法触发注册。
- **元数据状态机默认值错误**：`RuntimeMeta` 构造函数将 `ingestionStatus` 默认设为 `1` (Ingested)，这是导致“表面上看起来是完全且完美登记注册”（对应用户原话）的根源。
- **状态晋升断裂**：异步入库队列仅将根路径（对应用户原话：“拖拽某个项目到‘ArcMeta.Library_Z’文件夹里”）加入，导致子项永久卡死。
- **缓存污染**：`MftReader` 移动文件夹时不递归清理子项路径映射。
- **性能隐患**：在主线程执行同步物理磁盘递归扫描。

## 4. 详细解决方案

### 4.1 根除视觉欺骗 (修正默认值)
- 在 `src/meta/MetadataManager.h` 中，将 `RuntimeMeta` 构造函数中的 `ingestionStatus` 默认值从 `1` 修改为 `-2`（代表 Unknown/Pending 状态），以解决“实际却未成功登记注册”（对应用户原话）却显示成功的假象。

### 4.2 补全同盘移动入库逻辑 (重构准入校验)
- 修改 `AutoImportManager::onEntryUpdated`，在路径变更后增加 `isPathInManagedLibrary` 校验。
- **逻辑分支**：
    - 若 `oldPath` 为空且当前路径在库内：代表新移入项目，立即调用 `registerItem` 发起注册。
    - 若 `oldPath` 不为空且当前路径在库内：代表库内重命名，执行既有的重命名同步。

### 4.3 修复状态晋升链 (全量异步入库)
- 重构 `AutoImportManager::onEntryAdded`，将识别到的库内路径（含子项）批量推入 `m_pendingPaths` 队列，确保“拖拽项目”（对应用户原话）后所有文件都能晋升状态。
- 在 `processImportQueue` 中同步触发 `MetadataManager::registerItemsAsync` 执行深度解析。

### 4.4 路径缓存深度清理 (MftReader 优化)
- 在 `MftReader::updateEntryFromUsn` 处理重命名时，遍历 `m_path_cache` 并移除所有以旧路径为前缀的缓存项。

### 4.5 物理扫描异步化
- 使用 `QtConcurrent::run` 包装物理递归扫描，彻底解决导致系统假死的架构缺陷。

## 6. 实现准则与预警【核心】
- **依赖引用**：`AutoImportManager.cpp` 必须包含 `<QtConcurrent>` 以支持异步操作。
- **线程安全**：`MftReader` 的路径缓存清理操作必须被 `m_pathCacheMutex` 保护，严禁无锁操作。
- **信号处理**：批量注册完成后，应统一调用 `MetadataManager::notifyFullUIRebuild` 以防止 UI 刷新信号淹没。
