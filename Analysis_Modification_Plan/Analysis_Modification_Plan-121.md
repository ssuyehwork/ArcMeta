架构拨乱反正与规约落地方案 —— Analysis_Modification_Plan-121.md

## 1. 任务背景
针对此前由于 AI 脑补导致的“傻逼架构”——功能零散、重复造轮子、IO 扫描僭越等问题执行彻底整改。本方案旨在落实 `Development_Plan.md` 中确立的严谨分工，实现托管库自动化链路的闭环与手动干预逻辑的精准控制。

## 2. 问题定位

### 2.1 自动化链路断层
- **现象**：当前 `AutoImportManager` 绕过 `ImportHelper` 自行执行物理扫描，且缺乏明确的 0/1 标记位翻转逻辑。
- **根因**：对 `ReadDirectoryChangesW` 信号处理过于草率，试图以“全量补课”替代“增量登记”。

### 2.2 作用域感知缺失（傻逼逻辑）
- **现象**：重扫逻辑被误解为全库操作。
- **后果**：大规模无意义读盘导致 UI 假死与卡顿。

### 2.3 冗余轮子
- **现象**：`ImportHelper` 与 `AutoImportManager` 各有一套相似但不一致的物理扫描与数据库写入逻辑。

## 4. 详细解决方案

### 4.1 ReadDirectoryChangesW 驱动的身份登记 (Status 0)
- **修改位置**：`src/core/AutoImportManager.cpp` 中的 `onEntryAdded` 或对应监控回调。
- **核心逻辑**：
    - 捕获变动信号后，立即提取 **File_ID** 与 **Path**。
    - 调用 `MetadataManager::registerItem(path, true)`。
    - **强制标记**：在写入数据库时，显式设置 `ingestion_status = 0`（对应用户原话：“将项目的File_ID和传统路径注册登记到 metadata 表里并标记为0”）。

### 4.2 解析引擎对接与状态翻转 (Status 1)
- **修改位置**：`src/util/ImportHelper.cpp` (或合并后的摄取模块)。
- **核心逻辑**：
    - `ImportHelper` 接收状态为 0 的任务队列。
    - 执行实质性解析（提取颜色、尺寸等）。
    - **完工翻转**：解析成功后，调用专属函数将 `ingestion_status` 更新为 **1**（对应用户原话：“完成解析之后再将标记为0更新为1”）。

### 4.3 精准作用域“重新扫描”
- **修改位置**：`src/ui/ContentPanel.cpp` 右键菜单处理函数。
- **核心逻辑**：
    - **范围锁定**：获取 `view->selectionModel()->selectedIndexes()`（对应用户原话：“将被选中的部分执行重新强制扫描”）。
    - **强制指令**：无论目标项当前标记值为何，统一重置为 0 并推入 `ImportHelper` 解析序列（对应用户原话：“无论其标记值为0/1，都必须重新强制解析”）。

### 4.4 架构归一化：消除零散逻辑
- **物理动作**：将 `ImportHelper` 的核心函数迁移至 `AutoImportManager` 内部，或将两者统一重构为 `DataIngestionController`（对应用户原话：“两者不可重复造轮子”）。
- **禁令执行**：删除 `MetadataManager` 中的自动授权黑盒逻辑，确保每一笔入库数据均有迹可循。

## 5. 修改边界声明【红线】

**本次方案涉及范围：**
- [ ] `src/core/AutoImportManager.cpp`: 监控信号对接。
- [ ] `src/util/ImportHelper.cpp`: 状态翻转逻辑。
- [ ] `src/ui/ContentPanel.cpp`: 选中范围控制。

**明确禁止越界修改的范围：**
- [ ] 禁止对“ArcMeta.Library_”以外的路径执行自动化注册。
- [ ] 禁止在启动时执行任何递归读盘扫描。

## 6. 实现准则与预警【核心】
1.  **专属函数预警**：必须实现第 4 节提到的专属状态翻转函数，严禁在 UI 线程直接拼接 SQL 语句。
2.  **IO 分离**：`ImportHelper` 的解析任务必须在独立线程池运行，严禁阻塞 UI 刷新。
3.  **引用一致性**：确保 `ReadDirectoryChangesW` 上报的路径与数据库存储的路径格式 100% 对齐（正斜杠规范）。
