托管库逻辑归一化与架构清理方案 —— Analysis_Modification_Plan-120.md

## 1. 任务背景
针对当前版本中 `ArcMeta.Library_` 托管库在启动时违规自动扫描、右键菜单功能残缺、拖拽逻辑存在冲突以及代码实现高度冗余（ImportHelper 与 AutoImportManager 重复造轮子）等问题进行深度解构，并提供符合资深程序员标准的重构蓝图。

## 2. 问题定位

### 2.1 逻辑架构缺陷与“傻逼逻辑”
1.  **违规启动自扫描**：`AutoImportManager::syncAllManagedLibraries` 会在启动阶段强制触发全量递归入库。
2.  **右键菜单职责缺失**：托管库内项目（不论是否入库）缺乏显式的“重新扫描”触发入口。
3.  **拖拽交互漏洞**：`CategoryPanel` 对已入库项目的二次拖拽缺乏碰撞检测与用户反馈。
4.  **架构双轨制（重复造轮子）**：`ImportHelper` 与 `AutoImportManager` 存在显著的职能重叠，逻辑维护成本极高。

### 2.2 僵尸代码与冗余
1.  **分类项残留**：“未分类”系统分类项在当前业务逻辑下已无存在必要（对应用户原话：“‘未分类’可以彻底移除掉”）。
2.  **授权补丁冗余**：`MetadataManager` 中用于自动授权的逻辑补丁是导致自动扫描无法被手动控制的根源。

## 4. 详细解决方案

### 4.1 抑制启动自扫描
*   在 `AutoImportManager::syncAllManagedLibraries` 中，物理删除对 `handleRecursiveIngestion` 的调用（对应用户原话：“再次启动主程序时不该自动继续扫描入库”）。
*   启动流程改为仅识别托管库路径并确保 UI 树层级显示正确（对应用户原话：“只可以手动扫描入库”）。

### 4.2 增强右键菜单：持久化“重新扫描”
*   在 `src/ui/ContentPanel.cpp` 的菜单构建逻辑中，通过物理路径判定是否属于托管库（对应用户原话：“针对‘ArcMeta.Library_[盘符]’文件夹里的项目也要根据是否已入库来显示相应的选项”）。
*   **强制注入**：无论项目入库状态如何，均固定增加“重新扫描”菜单项（对应用户原话：“即便已经入库了，右键菜单中也要显示‘重新扫描’”）。

### 4.3 拖拽入库冲突拦截与提示
*   在 `src/ui/CategoryPanel.cpp` 的 `pathsDropped` 处理函数中，增加对 `ManagedRole` 的预校验。
*   **动作**：若检测到拖入的项目已标记为已入库，立即阻断后续注册流程（对应用户原话：“如果项目已经入库则弹出提示”）。
*   **弹窗**：调用 `FramelessMessageBox` 显示提示语：“该项目已入库，无需再次入库”（对应用户原话：“弹出提示‘该项目已入库，无需再次入库’”）。

### 4.4 彻底物理移除“未分类”项
*   从 `src/ui/CategoryModel.cpp` 的 `refresh` 函数中删除 `addSystemItem("未分类", ...)` 代码行（对应用户原话：“‘未分类’可以彻底移除掉”）。
*   同步清理相关统计更新逻辑。

### 4.5 架构合并：消除重复轮子
*   **废弃 ImportHelper**：物理删除 `src/util/ImportHelper.cpp`。
*   **归一化引擎**：将手动搬运与主动注册登记职能全部合并至 `AutoImportManager`（或重命名的归一化摄取控制器）（对应用户原话：“两者不可重复造轮子”）。
*   **授权硬化**：移除 `MetadataManager.cpp` 中的 1433-1445 行自动授权逻辑，确保入库动作必须由手动携带 `authorized=true` 触发。

## 6. 实现准则与预警【核心】
1.  **依赖预警**：重构后，所有 UI 层的入库请求必须显式传递 `authorized = true` 参数。
2.  **命名空间**：合并代码时必须保持 `ArcMeta` 命名空间的一致性。
3.  **头文件清理**：移除 `ImportHelper` 后，必须清理 `CategoryPanel.cpp` 等多处文件的 `#include` 残留，防止编译警告。
