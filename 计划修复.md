# ScanDialog C++ 移植修复计划：对标 FERREX-Rust 版本

## 一、问题背景

`ScanDialog.cpp` 由 FERREX-Rust 版本移植而来，但移植后存在多项功能缺失、性能退化和架构偏差。本文档逐项对比两个版本的差异，并给出具体修复方案。

---

## 二、逐模块差异对比与修复方案

### 🔴 模块 1：搜索引擎核心 (`search/lib.rs` → `MftReader::search`)

| 维度 | Rust 原版 | C++ 当前状态 | 严重程度 |
| :--- | :--- | :--- | :--- |
| **并行框架** | `rayon::into_par_iter` (工作窃取) | `QtConcurrent::blockingMappedReduced` | ⚠️ 中 |
| **字符串匹配** | `memchr::memmem` 字节流 + 手写 `contains_ignore_case` | `QString::contains` (UTF-16 框架开销) | 🔴 严重 |
| **属性过滤** | 过滤隐藏文件 (`0x0002`) 和系统文件 (`0x0004`) | **完全缺失**，`includeHidden`/`includeSystem` 字段声明了但未使用 | 🔴 严重 |
| **扩展名匹配** | 字节级比较，零分配 | `QString::mid` + `toLower` + `QStringList::contains`，每条目多次堆分配 | ⚠️ 中 |
| **空查询处理** | 空查询+无扩展名 → 直接返回空 | 空查询 → 返回**全部索引**（百万级），导致 UI 卡死 | 🔴 严重 |
| **路径搜索** | 仅搜索文件名 | 包含 `/` 或 `\` 时调用 `getFullPath`（极慢的递归路径回溯） | ⚠️ 中 |
| **前缀极速查找** | `search_prefix` 基于排序索引的二分法 `O(log n)` | **完全缺失** | 🔴 严重 |

#### 修复方案

1. **属性过滤**：在 `performRebuild` 的循环中，增加对 `FILE_ATTRIBUTE_HIDDEN (0x02)` 和 `FILE_ATTRIBUTE_SYSTEM (0x04)` 的检查
2. **空查询行为**：对标 Rust，空查询+无扩展名时直接返回空结果集
3. **字符串匹配优化**：将 `QString::contains` 替换为直接在 UTF-8 字节池上进行 `memchr` 风格的大小写不敏感匹配，避免每次构造 `QString`
4. **前缀查找**：在 `MftReader` 中维护一个排序索引 `m_sortedIndices`，实现 `searchPrefix` 方法，使用 `std::lower_bound` / `std::upper_bound`

### 🔴 模块 2：实时监控与覆盖层 (`spawn_usn_watcher` → `UsnWatcher` + Overlay)

| 维度 | Rust 原版 | C++ 当前状态 | 严重程度 |
| :--- | :--- | :--- | :--- |
| **覆盖层 (Overlay)** | `HashMap<u64, OverlayEntry>` 维护增删改，搜索时合并 | **完全缺失**，`applyChanges` 直接修改 SoA 数组 | 🔴 严重 |
| **搜索合并** | 搜索时：先查 SoA 索引，再合并 overlay 中匹配项 | 仅搜索 SoA，新增条目无法被即时搜到 | 🔴 严重 |
| **路径缓存** | `LruCache<u64, String>` (10000 条)，rename 时清空 | 无路径缓存，每次 `getFullPath` 递归回溯 | 🔴 严重 |
| **元数据补偿** | 搜索结果中 size=0 时，即时调用 `std::fs::metadata` 补偿 | **完全缺失**，MFT 扫描无法获取文件大小，所有文件显示 0 B | 🔴 严重 |
| **USN Reason 掩码** | `USN_REASON_DATA_OVERWRITE` | `USN_REASON_BASIC_INFO_CHANGE` | ⚠️ 中 |

#### 修复方案

1. **Overlay 机制**：在 `MftReader` 中新增 `std::unordered_map<uint64_t, OverlayEntry> m_overlay`，USN 事件写入 overlay 而非直接修改 SoA
2. **搜索合并**：`search()` 方法末尾追加 overlay 匹配逻辑
3. **LRU 路径缓存**：引入 `std::unordered_map` + 双向链表实现 LRU 缓存（或使用第三方 LRU 库），`getFullPath` 优先查缓存
4. **元数据补偿**：在 `ScanTableModel::data()` 的 `case 2 (大小)` 中，当 size=0 且非目录时，调用 `QFileInfo(path).size()` 补偿

### 🔴 模块 3：GUI 交互体验 (egui → Qt)

| 维度 | Rust 原版 | C++ 当前状态 | 严重程度 |
| :--- | :--- | :--- | :--- |
| **搜索触发** | 点击"搜索"按钮或 Enter 键触发 | `textChanged` 实时触发（每次按键都搜索） | ⚠️ 中 |
| **搜索历史** | 双击搜索框弹出历史列表，支持删除/清空 | **完全缺失** | ⚠️ 中 |
| **列排序** | 点击列头排序（名称/路径/大小/日期），支持升降序切换 | **完全缺失** | 🔴 严重 |
| **分页机制** | 每页 100 万条，搜索栏内置分页控件 | **完全缺失**，百万结果一次性加载到 QTableView | 🔴 严重 |
| **多选操作** | Ctrl+单击多选，Shift+单击范围选 | 仅 `ExtendedSelection` 模式（基础支持） | ✅ 基本对齐 |
| **选中统计** | 多选时状态栏显示：已选 X 项 \| 合计大小 \| 导出 CSV | 已实现选中统计，但缺少 CSV 导出 | ⚠️ 中 |
| **右键重命名** | 右键菜单 → 重命名 → 内联编辑文件名 | **完全缺失** | ⚠️ 中 |
| **盘符标签** | 每行结果前显示盘符标签 (如 `C:`) | **完全缺失** | ⚠️ 低 |
| **驱动器管理** | 右键驱动器：设为默认/忽略/恢复，配置持久化到 TOML | 仅支持"仅扫描此盘"，无持久化配置 | ⚠️ 中 |
| **NTFS 过滤** | 自动检测并仅显示 NTFS 卷 | 显示所有驱动器（包括非 NTFS） | ⚠️ 中 |
| **耗时显示** | 状态栏显示搜索耗时 (ms) | **完全缺失** | ⚠️ 低 |
| **数据占用** | 状态栏显示内存占用估算 | **完全缺失** | ⚠️ 低 |

#### 修复方案

1. **搜索触发优化**：将 `textChanged` 实时搜索改为防抖模式（300ms 延迟）或改为 Enter/按钮触发
2. **列排序**：在 `ScanTableModel` 中实现 `sort()` 重载，支持按名称/路径/大小/日期排序
3. **分页机制**：在 `ScanTableModel` 中维护 `m_currentPage` 和 `PAGE_SIZE`，`rowCount` 返回当前页条数
4. **CSV 导出**：新增 `exportSelectedCsv()` 方法
5. **搜索历史**：新增 `SearchHistoryPanel` 组件（项目中已有该文件）

### 🔴 模块 4：缓存持久化 (`storage/lib.rs` → `CacheManager`)

| 维度 | Rust 原版 | C++ 当前状态 | 严重程度 |
| :--- | :--- | :--- | :--- |
| **缓存格式** | FIDX (40字节/条目: frn+parent_frn+size+timestamp+name_offset+flags) | SCCH (40字节/条目，但**缺少 frn 和 parent_frn 字段**) | 🔴 严重 |
| **路径恢复** | 加载后通过 frn_map + parent_frn 链回溯完整路径 | 加载后 frn=0, parentFrn=0，**路径全部断裂** | 🔴 严重 |
| **USN 水位** | 持久化 `usn_watermark`，重启后从断点续监 | **完全缺失**，重启后丢失 USN 位置 | 🔴 严重 |
| **卷序列号** | 持久化 `volume_serial`，用于检测磁盘更换 | 已实现 | ✅ 对齐 |

#### 修复方案

1. **缓存格式升级**：`CacheEntry` 结构新增 `frn` 和 `parent_frn` 字段（与 Rust 的 `FileRecord` 对齐为 40 字节），版本号升至 2
2. **USN 水位持久化**：在 `CacheHeader` 中新增 `usn_watermark` 字段
3. **加载后恢复**：`loadFromCache` 后重建 `frn_to_idx` 和 `parent_indices` 映射

### ⚠️ 模块 5：MFT 扫描引擎 (`indexer/lib.rs` → `MftReader::performMftScan`)

| 维度 | Rust 原版 | C++ 当前状态 | 严重程度 |
| :--- | :--- | :--- | :--- |
| **缓冲区大小** | 64KB | 1MB | ✅ C++ 更优 |
| **NTFS 检测** | `GetVolumeInformationW` 过滤非 NTFS | **缺失**，尝试扫描非 NTFS 卷会失败 | ⚠️ 中 |
| **时间戳** | 使用 `record.TimeStamp` (FILETIME 格式) | 使用 `QDateTime::currentMSecsSinceEpoch()`（当前时间，非文件实际修改时间） | 🔴 严重 |
| **文件大小** | 均为 0（MFT 不含大小信息） | 均为 0 | ✅ 持平 |
| **FILE_FLAG_NO_BUFFERING** | 使用（减少系统缓存污染） | 未使用 | ⚠️ 低 |

#### 修复方案

1. **时间戳修复**：使用 `record->TimeStamp` 代替 `QDateTime::currentMSecsSinceEpoch()`，并转换 FILETIME → Unix 毫秒
2. **NTFS 检测**：扫描前调用 `GetVolumeInformationW` 检查文件系统类型

## 四、推荐实施策略

> [!IMPORTANT]
> 建议采用**彻底重构**策略，而非逐个打补丁。原因如下：
> 1. 缓存格式需要破坏性升级（新增 frn/parent_frn），旧缓存将不兼容
> 2. 搜索引擎需要从 `QString` 匹配全面切换到字节流匹配
> 3. Overlay 机制需要贯穿搜索、显示、更新全链路
> 4. 分页 + 排序需要重新设计 `ScanTableModel` 的数据流

### 重构顺序建议

```
阶段 1：数据层重构
  ├── 1.1 升级 CacheEntry 格式 (新增 frn, parent_frn, usn_watermark)
  ├── 1.2 修复 performMftScan 时间戳
  ├── 1.3 实现 LRU 路径缓存
  └── 1.4 实现 Overlay 机制

阶段 2：搜索引擎重构
  ├── 2.1 修复空查询行为
  ├── 2.2 实现属性过滤 (隐藏/系统)
  ├── 2.3 字节流匹配替代 QString
  ├── 2.4 元数据补偿 (文件大小)
  └── 2.5 实现前缀查找

阶段 3：UI 层重构
  ├── 3.1 分页机制
  ├── 3.2 列排序
  ├── 3.3 搜索防抖/Enter 触发
  ├── 3.4 搜索历史
  ├── 3.5 CSV 导出
  └── 3.6 状态栏完善 (耗时/内存)

阶段 4：配置与健壮性
  ├── 4.1 NTFS 卷检测
  ├── 4.2 驱动器配置持久化
  └── 4.3 USN 水位持久化
```
