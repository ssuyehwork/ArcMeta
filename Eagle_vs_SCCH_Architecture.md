# Eagle 与 SCCH (FRN) 元数据架构深度对比分析报告

## 1. 核心设计哲学对比

### Eagle: 资产包式管理 (Asset Bundle)
Eagle 采用的是**“强接管”**模式。它将每个素材封装在一个独立的 `.info` 文件夹中。
- **物理形态**：每一个素材都有一个乱码般的 ID 文件夹（如 `MAWE8JWJGTHS6.info`）。
- **元数据存储**：元数据（标签、评分、来源）以 `metadata.json` 的形式存放在该文件夹内部。
- **耦合度**：元数据与文件物理位置强绑定，但在文件系统层面具有一定的便携性（拷贝文件夹即拷贝元数据）。

### SCCH/ArcMeta: 非侵入式索引 (Non-Intrusive Index)
SCCH 采用的是**“旁路增强”**模式。它不改变用户原始文件的存储位置或命名。
- **物理形态**：文件保持原始名称（如 `提取主要颜色.md`）存放在用户自定义的路径下。
- **元数据存储**：通过集中的 `metadata.scch` 或目录下的 `.am_meta.json` 记录属性。
- **耦合度**：元数据与文件解耦，通过索引进行映射，保持了文件系统的可读性和开放性。

## 2. Eagle 关键 JSON 文件用途清单
针对 `Eagle/` 目录下的具体文件分析：

| 文件名 | 用途描述 | 角色 |
| :--- | :--- | :--- |
| **`metadata.json`** | 存储全局文件夹树、智能文件夹定义、快速访问列表及应用版本。 | **逻辑大脑** |
| **`mtime.json`** | 记录素材 ID 与其物理文件最后修改时间的映射表。 | **性能索引** |
| **`tags.json`** | 管理全局标签历史记录、搜索建议及星标常用标签。 | **标签池** |
| **`actions.json`** | 存储用户定义的自动化操作脚本（如导入时自动分类）。 | **自动化引擎** |
| **`saved-filters.json`** | 存储用户保存的复杂过滤/搜索条件预设。 | **搜索快照** |

## 3. 身份识别机制 (Identification)

| 特性 | Eagle (UID) | SCCH (FRN) |
| :--- | :--- | :--- |
| **主键类型** | 自生成随机字符串 (UID) | 文件参考号 (File Reference Number) |
| **追踪能力** | 仅限于软件内部移动 | 允许用户在资源管理器中直接移动 |
| **跨设备迁移** | 极佳（UID 是逻辑生成的） | 较弱（FRN 是文件系统底层的，跨分区会变） |
| **重命名容错** | 强（只要 UID 文件夹不动） | 极强（只要物理扇区上的文件没被删除重建） |

- **Eagle 的痛点**：如果你在软件外部移动了文件，Eagle 就会丢失关联，因为它的 UID 与物理路径的对应关系是存在它自己的数据库里的。
- **SCCH 的优势**：利用 NTFS/APFS 等现代文件系统的 **FRN**。即便你在 Windows 文件夹里随便拖动文件，应用再次扫描时能通过 FRN 瞬间识别出“它是原来那个带标签的文件”，而不需要重新标记。

## 4. 性能表现与 IO 模型

### Eagle: 并发小文件 IO
- **启动逻辑**：启动时需要并发读取成千上万个 `.info/metadata.json`。
- **优化方案**：使用 `mtime.json` 记录时间戳快照，实现秒级增量更新。
- **缺点**：在机械硬盘或高延迟文件系统上，大量随机读取小 JSON 性能下降明显。

### SCCH/ArcMeta: 集中式内存索引
- **启动逻辑**：一次性读入 `metadata.scch` 这样的大型 JSON 索引文件。
- **优化方案**：
    - **双重缓冲**：通过 60FPS 定时器平滑加载数据，不锁死主线程。
    - **渲染缓存**：针对图标颜色替换等 UI 渲染进行像素级缓存。
- **优点**：内存中查询速度极快，搜索所有带“商务”标签的素材几乎是瞬时的，无需遍历磁盘。

---

## 5. 适用场景总结

*   **Eagle (ID.info)**：适合**设计师的闭环灵感库**。它把素材“隔离”起来，防止误删，同时方便在不同电脑间通过云盘同步整个库，只要同步整个 ID 文件夹结构，数据就不会丢失。
*   **SCCH (FRN)**：适合**工程师或内容创作者的开放工作流**。它不强制要求你改变习惯，你可以继续用文件夹管理文件，它只是一层“智能增强层”，通过 FRN 默默地在底层为你维护标签和搜索索引。

---

**结论**：Eagle 的架构更偏向**“仓储式”**，安全但封闭；SCCH 的架构更偏向**“增强式”**，灵活且透明。
