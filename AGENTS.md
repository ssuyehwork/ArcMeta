# AGENTS.md — Jules 行为准则

> 本文件是 Jules 的强制行为规范。所有规则必须严格遵守，不得跳过、简化或以"效率"为由绕过任何步骤。

---

## 第一条：任务接收 — 逐字阅读，不得跳读

收到用户消息后，Jules 必须：

1. **完整阅读用户的每一句话**，包括括号内的补充说明、代码注释引用、以及任何看似"次要"的描述。
2. **不得基于消息的前几句话就开始假设任务意图**。用户的真实意图往往在消息末尾或中段才完整呈现。
3. 如果消息超过三句话，**禁止在阅读完成之前启动任何代码修改计划**。

---

## 第二条：意图复述 — 必须在动手之前完成

在进行任何文件修改、代码生成、或方案制定之前，Jules **必须**先用自己的话复述用户的意图，格式如下：

```
【我的理解】
用户想要：<用一到三句话概括核心目标>
涉及的文件/模块：<列出相关文件>
不应触碰的范围：<列出用户明确或隐含禁止修改的部分>
我的疑问（如有）：<列出任何不确定之处>
```

**只有在用户确认理解正确之后，才可以开始执行。**

若用户没有明确回复"理解正确"或等价确认，Jules 不得视沉默为同意而自行推进。

---

## 第三条：禁止擅自扩大修改范围

Jules 只能修改用户明确指定的内容。严格禁止以下行为：

- 以"顺手优化"、"物理对齐"、"统一风格"等理由修改用户未提及的代码
- 删除、注释、或移动任何用户未提及的函数、成员变量、布局参数
- 将 `margin`、`spacing`、`padding` 等数值归零并声称"更合理"
- 恢复已被用户删除的类或组件（如 `StarRatingWidget`、`ColorPickerWidget`）
- 修改构造函数初始化顺序、事件过滤器安装时机等具有隐性依赖的结构

**每一处修改都必须能在用户的原始消息中找到明确的对应依据。**

---

## 第四条：修改记录 — 强制写入 Modification_Record.md

每次执行完任务后，Jules 必须在 `Modification_Record.md` 中追加记录，格式如下：

```
## [YYYY-MM-DD HH:MM:SS]
### 任务描述
<用户的原始需求，一句话概括>

### 修改的文件
- `文件路径`：<修改内容描述>

### 未修改的文件
- <明确列出哪些相关文件被评估但未动>

### 潜在风险
- <列出此次修改可能引入的副作用>
```

此记录为强制项，不得遗漏。

---

## 第五条：高风险操作必须提前申报

以下操作在执行前必须单独列出并等待用户批准：

| 高风险操作类型 | 示例 |
|---|---|
| 修改类的继承关系或接口 | 改变基类、添加纯虚函数 |
| 修改构造/析构顺序 | 调整成员初始化列表 |
| 修改信号槽连接方式 | 从直连改为队列连接 |
| 删除或重命名公开成员 | 删除 `public:` 方法或属性 |
| 修改数据库 Schema | 增删字段、修改表结构 |
| 修改事件过滤器的安装/卸载时机 | `installEventFilter` 调用位置变动 |
| 修改任何布局的 margin/spacing 参数 | 即使原值看起来"不合理" |

申报格式：

```
【高风险操作申报】
操作：<描述>
原因：<为何需要此操作>
风险：<可能的副作用>
请确认是否执行：是 / 否
```

---

## 第六条：禁止伪造理由

Jules 不得使用以下类型的措辞来为未经授权的修改辩护：

- "物理对齐要求"
- "更符合 Qt 规范"
- "原代码存在潜在问题"
- "顺便修复了一个隐患"
- "统一了代码风格"

**若修改没有对应用户需求，任何理由都不成立。**

---

## 第七条：不理解时，必须提问而非猜测

当用户的需求存在歧义时，Jules 必须：

1. 明确列出歧义点
2. 提出具体问题请用户澄清
3. **禁止"猜一个最可能的意思"然后直接执行**

宁可多问一次，也不能引入一次错误的修改。

---

## 附：Jules 每次任务的标准执行流程

```
1. 完整阅读用户消息（禁止跳读）
       ↓
2. 输出【我的理解】复述意图
       ↓
3. 等待用户确认
       ↓
4. 检查是否涉及高风险操作 → 如有，申报并等待批准
       ↓
5. 仅修改用户明确指定的内容
       ↓
6. 写入 Modification_Record.md
       ↓
7. 汇报完成，列出所有变更
```

**违反以上任意一条规则，均视为本次任务执行失败。**

# 修改代码必须严格遵守以下防护规则：  
- 修改时必须结合上下文，保持函数签名一致。  
- 禁止调用不存在的函数、类或命名空间。  
- 禁止随意增加或减少参数。  
- 禁止使用已弃用成员。  
- 禁止脑补或重构未授权的部分。  
- 所有修改必须结合上下文，确保逻辑闭环，做到开箱即用。
- 修改后必须通过静态检查，禁止出现以下典型的编译错误：
    - 调用不存在的成员函数（如在不支持的 Qt 版本或类中调用 `setSelectionRectVisible`）。
    - 变量名冲突（如“type”声明隐藏了上一个本地声明）。
    - 使用未声明或未定义的标识符/宏（如 `TRASH_CATEGORY_ID`）。
    - 变量初始化顺序错误。
    - 函数参数类型不匹配（如将 `int` 传给 `std::vector` 类型的参数）。
- **系统 ID 物理一致性**：侧边栏系统项 ID 与 `CategoryRepo` 常量必须物理对齐（如 Trash=-8, Uncategorized=-2）。
- **UI 视觉规范**：右键菜单必须使用 `UiHelper::applyMenuStyle` 以确保样式统一，严禁圆角外出现直角溢出。

# 1. 用户需求

## 1.0 元数据面板（MetaPanel）物理清理：彻底移除“星级评定”与“颜色标记”UI区域。为确保外部兼容性，必须保留 `setRating`、`setColor` 公共接口及 `metadataChanged` 信号声明，但其实作必须为空。
## 1.0.1 元数据面板（MetaPanel）视觉修复：恢复面板左右两侧的物理边距（10px），严禁将右侧边距设为 0px 以防止内容紧贴容器边缘或被截断。

## 1.1 "最近访问"是应用内行为，不依赖 Windows 的 ftLastAccessTime，而是用户在界面里打开/读取了哪些文件，由应用自己记录。例如，选中项目并按下空格键预览、重命名、变更等等操作，均视为访问
## 1.2 选中项目并按下空格键，应该有个判断逻辑，例如，如果选中的项目是文件夹、exe程序等不支持编辑的项目则无需执行预览，甚至不必要打开预览界面，直接执行Return即可
## 1.3 侧边栏分类计数逻辑：杜绝采用“...”占位符。如果统计尚未完成需要显示占位符，则必须统一显示为“(0)”。
## 1.4 回收站联动逻辑：一旦数据（File ID / FRN）被移入回收站，其所属的所有分类（包括“全部数据”、“未分类”以及用户自定义分类）的计数必须相应扣减。
## 1.5 侧边栏交互：禁用 Ctrl+A 全选功能，防止用户误操作选中所有分类。
## 1.6 回收站右键菜单：回收站应具有专属菜单，包含“清空回收站”和“还原全部项目”两个核心功能。
## 1.7 内容区框选样式：内容容器在进行左键拖动框选时，必须显示蓝色透明的框选视觉效果。
## 1.8 右键菜单指示符：必须确保右键菜单正确显示子菜单指示符（实心三角形），以便用户识别包含子选项的主项。样式需统一使用 `UiHelper::applyMenuStyle` 维护，指示符应采用物理临时文件（通过 `getSvgTempFilePath`）以确保渲染稳健性，并配合 `subcontrol-origin: padding` 与 `right: 8px` 进行靠右边缘定位。
## 1.9 菜单删除逻辑：移除“彻底删除”选项，将“安全擦除”重命名为“永久删除”。
## 1.10 扫描入库逻辑：选中文件夹或非图像文件时显示“扫描入库”选项。
## 1.11 自动入库逻辑：对未入库项执行分类操作时，系统必须自动先执行入库登记流程。
## 1.12 地址栏增强：集成刷新按钮、F5 快捷键支持、双击显示历史路径面板及历史记录持久化（上限 15 条）。
## 1.13 撤销/重做机制：建立基于命令模式的双栈撤销系统，支持 Ctrl+Z 和 Ctrl+Shift+Z 快捷键，涵盖重命名、移动、星级/颜色变更及分类关联。
## 1.14 UI 物理对齐：导航栏搜索框宽度固定为 230px，且其右侧禁止添加额外间距，以确保与右侧筛选面板完全对齐。
## 1.15 边缘缩放机制：`ResizeEventFilter` 必须全局安装在 `QCoreApplication` 上，以拦截子控件上的鼠标事件并实现无边框窗口的边缘缩放。
## 1.16 初始化顺序护栏：`MainWindow` 的事件过滤器（`m_hoverFilter`, `m_resizeFilter`）必须在 `initUi()` 之前创建。严禁使用 `if` 静默保护，必须使用 `Q_ASSERT` 确保顺序正确，防止功能静默失效。
## 1.17 地址栏焦点反馈：由于 QSS 不支持 `:focus-within`，地址栏容器的焦点蓝色边框必须在 `AddressBar::eventFilter` 中通过监听子控件的 `FocusIn/FocusOut` 事件，并手动更新动态属性及样式（`unpolish/polish`）来实现。
## 1.18 崩溃现场证据捕获机制：集成 WER 完整 Dump 配置脚本，确保日志处理器 (`customMessageHandler`) 执行 `flush` 强制落盘，并在 `MainWindow` 中实现自动化资源监控（每 5 秒记录 GDI/USER 句柄及内存占用），以便在不依赖外部工具的情况下精准锁定“罪证”。
## 1.19 导入逻辑归一化：将“扫描入库”与“拖拽导入”的行为逻辑完全对齐。必须建立统一的函数模块（如 `ImportHelper`）来执行“递归扫描+元数据注册+分类镜像创建”流程。
    - **UI反馈**：进度对话框必须实时反馈处理动态，格式统一为：`[当前/总数] 百分比% - 当前处理的文件名`。
    - **目录镜像**：当从内容区执行“扫描入库”时，其镜像分类节点应始终创建在“我的分类”的最顶层（根目录）。
    - **批次事务**：数据库写入必须从“单条提交”优化为“大事务分批提交”，规定每处理 500 项执行一次 `COMMIT`，以兼顾 I/O 性能与崩溃后的数据一致性。
    - **中断保护**：若用户中途关闭进度框，必须弹出二次确认对话框，询问用户是“保留当前已入库进度”还是“执行一键撤销并退出”。
    - **撤销机制**：利用 `UndoManager` 建立导入任务的快照，支持通过 `Ctrl+Z` 一键清理整个导入任务产生的元数据与分类关联。
## 1.20 单实例运行限制：主程序必须实现单实例检测机制，防止用户开启多个应用实例导致数据库写入冲突或系统资源异常占用。
## 1.21 退出流程优化与文件占用解决：彻底解决从托盘退出后进程（ArcMeta.exe）残留及数据库文件被占用的故障。
    - **退出反馈**：禁止在析构函数中执行耗时的同步 `flushAll`。退出时须弹出“正在安全保存数据...”的模态进度提示。
    - **步进式备份**：持久化逻辑必须改用步进式 `sqlite3_backup_step`（如每 50 页一跳），并在跳跃间隙执行事件循环处理（`processEvents`）以维持界面响应并提供进度反馈。
    - **强制中断机制**：所有后台耗时线程（如 USN 扫描、MFT 读取）必须支持 `AtomicBool` 中断位，确保在退出指令下达后能立即安全停止。
    - **物理句柄清理**：确保在进程正式结束前，显式调用 `sqlite3_close_v2` 关闭所有数据库句柄，解除对物理文件的占用。
## 1.22 筛选器增强：
    - **快速输入**：在“颜色标记”、“标签/关键字”、“文件类型”、“创建日期”和“修改日期”分类标题下方新增 `QLineEdit` 快速输入框（排除“评级”）。
    - **触发逻辑**：输入文字不立即触发过滤，按下回车键（Enter）后执行。支持模糊匹配，并与下方复选框状态保持“且”（AND）关系。
    - **历史面板**：双击输入框弹出 `SearchHistoryPanel`（复用现有组件），每个分类独立记录最近 10 条历史，支持点击填入及删除。
    - **联动重置**：点击筛选面板顶部的“清空”（垃圾桶图标）时，同步清空 5 个输入框的文字。
## 1.23 重命名与刷新增强：
    - **智能选中**：执行重命名（F2/右键菜单）进入编辑状态时，系统必须自动识别文件扩展名。若是文件，默认仅选中名称部分（不含后缀）；若是文件夹或分类，则全选名称。
    - **联动刷新**：重命名成功或执行“解析颜色”提取色板后，必须立即触发 `onSelectionChanged` 信号，强制驱动元数据面板（MetaPanel）同步刷新显示最新的名称、路径或颜色标记数据。

## 1.24 文件夹入库进度标识 (Plan-27)：
    - **进度感知**：将文件夹的“已入库”状态从二元升级为进度感知模式。
    - **UI表现**：在卡片右上角以蓝色圆弧进度环形式呈现入库百分比。
    - **交互**：鼠标悬停在进度环上时，通过 ToolTipOverlay 显示具体百分比。

## 1.25 颜色解析算法精度优化 (Plan-28)：
    - **感知加权**：引入“感知显著性”加权模型，对近似白色背景进行强力降权，对暗部细节进行提权。
    - **相似色抑制**：引入 CIELAB Delta E 算法，收紧合并阈值并增强空间排斥逻辑，确保提取的色板更具层次感和多样性。

## 1.26 线程安全与稳定性加固 (Plan-28)：
    - **COM初始化**：确保所有涉及 Windows Shell API 调用的后台线程（导入、扫描、解析）均执行 `CoInitializeEx`，解决 Shell 接口返回异常导致的颜色丢失问题。
    - **重试机制**：改良视觉元数据重试队列的移除策略，仅在处理成功或确认无法提取时移除。

## 1.27 图片尺寸自动识别与多维筛选 (Plan-29, Plan-30)：
    - **自动记录尺寸**：在扫描入库过程中自动提取并记录图片的像素宽高，支持 PSD/AI 等专业格式。
    - **多维筛选增强**：
        - **链接/备注**：支持筛选“有/无链接”或“有/无备注”的文件。
        - **大小筛选**：支持输入数字范围并自动进行 KB/MB/GB 单位换算的物理大小筛选。
        - **比例筛选**：基于提取的尺寸，支持横图、竖图、方形及 16:9 比例的逻辑筛选。

# 2. 交互与回复规范

## 2.1 语言规范：Jules 在回复用户、总结工作或进行代码审查回复时，必须始终使用中文。
