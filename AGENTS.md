## 2. 修改记录 "Modification_Record.md" 文件
## 强制要求：代码修改必须全量记录到 Modification_Record.md

你每次修改、新增或删除任何代码行，都必须同步将变更记录写入项目根目录下的 `Modification_Record.md` 文件。这是不可跳过的硬性要求，优先级高于一切其他任务。

---

### 记录格式规范

每次变更追加一条记录块，格式严格如下：

---
## [变更序号] 变更时间：YYYY-MM-DD HH:MM:SS（精确到秒）

**文件路径：** `相对路径/文件名.扩展名`
**变更类型：** 修改 / 新增 / 删除（三选一）

### 修改前（Before）
```语言
此处粘贴修改前的原始代码，一行不漏
```

### 修改后（After）
```语言
此处粘贴修改后的最终代码，一行不漏
```

### 变更说明
- 变更原因：（简述为何要改，不得为空）
- 影响范围：（列出受影响的函数/模块/接口）
- 是否在需求范围内：是 / 否（超出需求范围的修改严禁执行）

---

### 各变更类型的特殊规则

**修改（Modify）：**
- 必须同时写出"修改前"和"修改后"两段完整代码块
- 不得仅记录 diff，必须是可直接复制还原的完整代码

**新增（Add）：**
- "修改前"区块填写：`（此处原为空，以下为新增内容）`
- "修改后"区块写出所有新增代码

**删除（Delete）：**
- "修改前"区块写出被删除的完整代码
- "修改后"区块填写：`（已删除，此处无代码）`

---

### 禁止行为

1. 禁止在未记录到 Modification_Record.md 的情况下提交任何代码变更
2. 禁止将多次独立变更合并为一条记录——每个文件的每次变更单独一条
3. 禁止修改超出用户明确要求范围之外的任何代码
4. 禁止以"顺手优化""顺带修复"为由擅自改动需求外代码
5. 禁止省略"变更说明"中的任何字段
6. 时间戳必须使用实际操作时间，禁止使用占位符或估计时间

---

### 执行顺序

1. 接收用户需求
2. 确认变更范围，若涉及需求外代码，必须暂停并征得用户明确同意
3. 执行代码变更
4. 立即（同一步骤内）将本次变更完整写入 Modification_Record.md
5. 告知用户"已记录到 Modification_Record.md，变更序号为 [N]"

如果 Modification_Record.md 不存在，在第一次记录前自动创建该文件。

## 3. 【强制规则 A】动任何代码之前，必须先完成分析并写入 Analysis_Modification_Plan.md

本规则与 Modification_Record.md 规则并列为最高优先级。
执行顺序：先写分析计划 → 用户审批 → 再动代码 → 记录变更。
跳过分析计划直接修改代码，视为严重违规。

  ✗ "修改 X 函数使其正常工作"
  ✗ "优化代码逻辑"

  ✗ "改善性能"
  ✗ 任何不包含具体技术路径的模糊描述
