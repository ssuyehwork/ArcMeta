# ArcMeta/FERREX 数据持久化架构分析报告 (Analysis_Modification_Plan-21.md)

本报告深入拆解了系统中 **DB 模式 (SQLite)** 与 **JSON 模式** 的技术细节、文件结构及逻辑共存机制。

---

## 1. 两种模式的核心逻辑架构

### 1.1 DB 模式 (中心化加速层)
*   **物理形式**：`arcmeta_main.db` (SQLite 3)。
*   **核心逻辑**：作为应用运行时的“高性能引擎”。利用数据库的索引机制、事务处理和关联查询（JOIN）来实现秒级的分类统计和最近使用记录检索。
*   **适用场景**：侧边栏分类树加载、复杂标签过滤、关联文件 ID 的快速定位。

### 1.2 JSON 模式 (分布式真值层)
*   **物理形式**：一系列离散或集中的 `.json` 文件（详见第二章）。
*   **核心逻辑**：作为数据的“真值来源 (Source of Truth)”和“可移植镜像”。JSON 模式的存在确保了即使数据库损坏，用户的数据（分类、星级、颜色）依然物理存在于文件夹中。
*   **适用场景**：跨设备同步、数据备份、外部资源管理器预览。

---

## 2. JSON 模式包含哪些具体文件？

根据代码审计，JSON 模式并非单一文件，而是一个**分层文件体系**：

| 文件名称 | 存放位置 | 包含的具体内容 | 逻辑用途 |
| :--- | :--- | :--- | :--- |
| **`arcmeta_categories.json`** | 程序根目录 | 完整的虚拟分类树结构、父子关系、分类颜色、排序权重。 | 存储侧边栏的“分类文件夹”定义。 |
| **`.am_meta.json`** | **各个物理文件夹内** | 该目录下所有文件的颜色标记、星级、标签、备注、FRN 镜像。 | **分布式存储**。文件移动时元数据随文件夹走，实现“数据跟随”。 |
| **`FERREX_drivers.json`** | 程序根目录 | 物理磁盘卷序列号、驱动器挂载点、磁盘特定的元数据。 | 解决不同电脑盘符变动导致的元数据错位问题。 |
| **`All_FRN_am_meta.json`** | 程序根目录 | 记录了所有离散 `.am_meta.json` 的物理 FRN (File Reference Number)。 | **内存对账**。系统通过此文件快速定位全盘有哪些目录包含元数据，无需全盘扫描。 |
| **`metadata.json`** | `FERREX_Category/` | 分类体系描述（参考 Eagle 格式）。 | 用于定义虚拟文件夹的高级属性。 |

## 3. 两种模式的共存与同步机制

系统采用了一种 **“双轨制持久化”** 策略，代码层面的体现如下（以 `CategoryRepo.cpp` 为例）：

1.  **同步写入逻辑**：
    ```cpp
    bool CategoryRepo::add(Category& cat) {
        bool jsonOk = JsonCategoryEngine::add(cat); // 先写 JSON
        bool dbOk = db_insert(cat);                 // 再写 SQLite
        return m_isJsonMode ? jsonOk : dbOk;
    }
    ```
    这种逻辑确保了数据在磁盘 JSON 文件和数据库中是**实时镜像**的。

2.  **真值优先级分流**：
    *   当 `m_isJsonMode = true` 时，UI 加载完全忽略数据库，直接从 `arcmeta_categories.json` 解析。
    *   当 `m_isJsonMode = false` 时，系统优先从数据库读取，仅在数据库缺失时通过“同步引擎”从离散的 `.am_meta.json` 中重建索引。

3.  **冲突处理（内存对账）**：
    `All_FRN_am_meta.json` 扮演了“裁判”的角色。它记录了所有 JSON 文件的物理身份（FRN），当数据库记录与物理文件不符时，系统以物理文件中的 JSON 数据为准。

## 4. 架构评价与建议

### 4.1 优点
*   **极高的容灾能力**：数据库挂了没关系，`.am_meta.json` 还在，数据就在。
*   **物理隔离性**：通过在 ContentPanel 等 UI 组件中过滤 `.am_meta.json`，实现了对用户透明的静默管理。

### 4.2 潜在风险
*   **I/O 频率过高**：每次修改标签都会触发物理文件的重写。
*   **建议**：在 `Analysis_Modification_Plan-20.md` 中提到的“混合动力架构”正是为了优化这一点——**高频修改进 DB，关键节点存 JSON**。
