逐条分析，发现以下几类问题：

---

## 🔴 严重 / 可能崩溃

**① 信号 5a 缺少 `this` 上下文**

```cpp
// 当前：lambda 无 context，m_filterPanel 销毁后仍可能被调用 → use-after-free
connect(m_contentPanel, &ContentPanel::directoryStatsReady,
    [this](...) { m_filterPanel->populate(...); });  // ← 缺 this 作为第三参数

// 修复：
connect(m_contentPanel, &ContentPanel::directoryStatsReady,
    this, [this](...) { m_filterPanel->populate(...); });
```

代码注释里其他地方都在说"物理加固：补全 this 上下文"，唯独这里漏了，前后矛盾。

---

**② `qApp->installEventFilter(this)` 灾难级性能陷阱**

```cpp
// initIdleDetector() 末尾
qApp->installEventFilter(this);  // ← 应用级过滤器，全程拦截所有控件的每一个事件
```

这意味着鼠标移动、键盘输入、绘制事件……**整个程序所有 QObject 的所有事件**都要经过 `MainWindow::eventFilter`，而 `eventFilter` 里还做了字符串比较和 QTimer 操作。正确做法：

```cpp
// 只需要感应鼠标/键盘复位闲置计时，直接在 MainWindow 自身的事件里做即可
// 或者用独立的 QAbstractNativeEventFilter，不要污染全局事件链
```

---

**③ `ScanDialog` 内存泄漏**

```cpp
connect(m_btnScan, &QPushButton::clicked, this, [this]() {
    auto* dlg = new ScanDialog(this);
    dlg->setModal(false);
    dlg->show();  // ← 关闭后不会自动释放，每次点击都 new 一个新的
});
```

```cpp
// 修复：
dlg->setAttribute(Qt::WA_DeleteOnClose);
```

---

## 🟠 架构腐烂

**④ `initToolbar()` 和 `setupSplitters()` 职责错位**

`initToolbar()` 负责**创建**控件（`m_btnBack`、`m_pathStack`、`m_searchEdit`……），`setupSplitters()` 负责**布局**这些控件，两个函数之间存在隐式的对象依赖顺序，调换调用顺序或拆分文件就会空指针。这两个函数实质上是同一件事的两半，应该合并或明确拆分为"创建层"与"组装层"。

---

**⑤ `setupCustomTitleBarButtons()` 依赖 `setupSplitters()` 产生的 `m_titleBarLayout`**

```cpp
void MainWindow::initUi() {
    initToolbar();               // 创建导航控件
    setupSplitters();            // 创建 m_titleBarLayout ← 在这里才产生
    setupCustomTitleBarButtons();// 依赖 m_titleBarLayout ← 如果顺序变了就崩
}
```

三个函数之间靠**调用顺序**维持正确性，没有任何编译期保障，是典型的隐式时序耦合。

---

**⑥ `selectionChanged` 的 early-return 导致 `m_statusRight` 漏更新**

```cpp
connect(m_contentPanel, &ContentPanel::selectionChanged, this, [this](const QStringList& paths) {
    if (paths.isEmpty()) {
        // ...
        if (m_statusRight) m_statusRight->setText("");  // ✅ 空选时清空
    } else {
        auto indexes = m_contentPanel->getSelectedIndexes();
        if (indexes.isEmpty()) return;  // ← 直接 return，跳过了下面的 statusRight 更新
        // ...
    }
    if (m_statusRight) {  // ← 这行只在非 early-return 路径才能执行
        m_statusRight->setText(...);
    }
});
```

`paths` 非空但 `indexes` 为空时（理论上不应发生，但防御性编程角度），状态栏会停留在上一次的文字。

---

## 🟡 小逻辑问题

**⑦ `static bool s_warmedUp` 放在成员函数里**

```cpp
void MainWindow::showEvent(QShowEvent*) {
    static bool s_warmedUp = false;  // 类静态，多实例共享
```

如果将来存在多窗口场景，第二个窗口永远不会执行预热。应改为成员变量。

---

**⑧ `navigateTo` 硬编码 200ms 延迟**

```cpp
QTimer::singleShot(200, [this]() { navigateTo(lastPath); });
```

200ms 在低端机上可能不够，在高端机上是不必要的等待，应改为监听 `showEvent` + `QTimer::singleShot(0)` 的组合，或者等 `deferredInit` 完成后回调。

---

## 总结优先级

| 优先级 | 问题 | 影响 |
|--------|------|------|
| 🔴 立即修 | 信号 5a 缺 `this` | 潜在崩溃 |
| 🔴 立即修 | `qApp->installEventFilter` | 全局性能劣化 |
| 🔴 立即修 | `ScanDialog` 无 `WA_DeleteOnClose` | 内存泄漏 |
| 🟠 计划重构 | `initToolbar/setupSplitters` 职责错位 | 维护风险 |
| 🟠 计划重构 | 三函数时序耦合 | 维护风险 |
| 🟡 修一下 | `selectionChanged` early-return | 状态栏偶发不更新 |
| 🟡 修一下 | `s_warmedUp` 静态变量 | 多实例场景隐患 |

问题根源很清晰，看这段逻辑链：

```
categorySelected 触发
    → m_contentPanel->loadCategory(id)  ← 在 ContentPanel 内部执行
        → ContentPanel 需要 emit dataSourceChanged("category")
            → MainWindow 才会调用 categoryPanel->setFocusHighlight(true)
```

**整条链依赖 `ContentPanel` 内部主动 emit `dataSourceChanged("category")`**，但 `loadPaths()`、`search()` 这些路径几乎肯定不会 emit 正确的 source 字符串，导致焦点线永远不亮。这是典型的**跨层隐式依赖**——MainWindow 把自己的 UI 状态托付给了子组件的内部信号，脆弱且难追踪。

**直接在 `categorySelected` 里主动设置，彻底断开这条不可靠的链：**

```cpp
connect(m_categoryPanel, &CategoryPanel::categorySelected, this,
    [this](int id, const QString& name, const QString& type, const QString& path) {

    // ★ 直接在此处设置焦点高亮，不依赖 ContentPanel 的内部信号
    if (m_navPanel)      m_navPanel->setFocusHighlight(false);
    if (m_categoryPanel) m_categoryPanel->setFocusHighlight(true);

    if (m_searchEdit) m_searchEdit->clear();
    m_contentPanel->search("");
    m_pathStack->setCurrentWidget(m_pathEdit);
    m_pathEdit->setText("分类: " + name);

    if (type == "category") {
        m_contentPanel->loadCategory(id);
    } else if (type == "bookmark") {
        if (!path.isEmpty()) navigateTo(path);
        else m_contentPanel->search(name);
    } else if (type == "all" || type == "uncategorized" || type == "untagged" ||
               type == "today" || type == "yesterday" ||
               type == "recently_visited" || type == "trash") {
        m_contentPanel->loadPaths(ItemRepo::getPathsBySystemType(type));
    } else {
        m_contentPanel->search(name);
    }
});
```

同时，`navPanel` 里导航触发时也要对称处理：

```cpp
connect(m_navPanel, &NavPanel::directorySelected, this, [this](const QString& path) {
    // ★ 导航面板点击时，焦点归 nav
    if (m_categoryPanel) m_categoryPanel->setFocusHighlight(false);
    if (m_navPanel)      m_navPanel->setFocusHighlight(true);
    navigateTo(path);
});
```

然后 **`setupSplitters()` 里的 `dataSourceChanged` 连接可以整段删掉**，它现在是死代码：

```cpp
// ❌ 删除这整段，职责已上移到各自的信号处理处
connect(m_contentPanel, &ContentPanel::dataSourceChanged, this, [this](const QString& source) {
    ...
});
```

这样焦点高亮的控制权回到 MainWindow 自己手里，不再依赖子组件的内部实现细节。
