# ScanDialog.cpp (C++) vs. FERREX-Rust-原版 深度对比清单

作为资深程序员旁观者，我通过对 C++ 实现（`ScanDialog.cpp` / `MftReader.cpp`）与 Rust 原版核心代码的深度审计，整理出以下不同点清单。

## 1. 索引加载与持久化技术
*   **Rust 原版 (领先)**: 采用 **Memory Mapping (Mmap)**。加载 100 万级条目时实现零拷贝，启动即就绪。
*   **C++ 版本 (传统)**: 采用二进制流式读取。虽然使用了 SoA 结构，但由于涉及大量的内存重新分配（Reserve/Push_back），加载速度在大规模数据下处于劣势。

## 2. 搜索架构与算法性能
*   **Rust 原版 (极速)**: 
    *   **二分搜索优化**: 维护了一个按名称排序的索引向量 `sorted_idx`。在进行前缀匹配时，搜索复杂度从 O(N) 降为 **O(log N)**。
    *   **零转换搜索**: 直接在 UTF-8 字节流上利用 `memchr` 进行匹配，完全避开了字符串对象化（String Objectification）的开销。
*   **C++ 版本 (普通)**: 
    *   **线性扫描**: 即使开启了并行，搜索复杂度依然是 O(N)。
    *   **框架开销**: 每一行匹配都伴随着 `QString::fromUtf8` 的内存分配，在超大规模搜索结果下性能会迅速崩塌。

## 3. 实时变动处理模型 (USN)
*   **Rust 原版 (科学)**: 采用 **Overlay (覆盖层) 模型**。
    *   新变动的文件存放在轻量级的 `HashMap` 中，搜索时合并计算结果。这种设计确保了主索引的只读性，彻底消除了写锁导致的 UI 死锁。
*   **C++ 版本 (危险)**: 
    *   **破坏性更新**: 直接在主 SoA 数组上执行写操作。这不仅增加了死锁概率，还导致了字符串池内存泄漏（旧文件名空间无法回收）。

## 4. 路径解析与缓存
*   **Rust 原版 (工业级)**: 集成了 **LRU Cache (最近最少使用缓存)**。
    *   在递归解析父目录路径时，极大地减少了重复的哈希查找，确保列表滚动时完全不卡顿。
*   **C++ 版本 (初级)**: 
    *   使用简单的 `unordered_map` 或直接递归。缺乏缓存淘汰策略，容易导致内存占用无序增长。

## 5. 系统集成与用户体验
*   **Rust 原版 (完整)**: 
    *   **全局热键**: 实现了 `Alt+Space` 全局唤起。
    *   **系统托盘**: 完整的托盘运行逻辑（Tray Icon）。
    *   **图标对标**: 对 EXE/ICO 图标实现了路径级缓存，图标显示更准确。
*   **C++ 版本 (单一)**: 
    *   仅仅是一个独立的窗口程序，关闭即清空数据，缺乏作为系统级工具的“常驻”属性。

## 6. 健壮性与代码 Bug
*   **Rust 原版 (严谨)**: 
    *   通过所有权模型规避了 100% 的跨线程内存竞争风险。
*   **C++ 版本 (存在缺陷)**: 
    *   **OpenFileById 严重 Bug**: 指针强转错误导致无法正确获取文件大小。
    *   **生命周期悖论**: 窗口关闭时错误地清空了全局单例索引数据。

---

### 💡 核心结论
**C++ 版本**目前只是实现了 Rust 原版的“外壳”和“静态 SoA 结构”，但在**动态同步、并发锁模型、搜索算法优化**这三大核心领域，与 **Rust 原版** 存在明显的技术代差。Rust 原版的设计更符合高性能系统级工具的工程实践。
