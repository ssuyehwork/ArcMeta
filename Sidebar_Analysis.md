# 侧边栏分类系统对比分析报告：ArcMeta vs Eagle

## 1. 数据架构与存储 (Data Architecture)
*   **Eagle (JSON/UID 驱动)**:
    *   **中心化索引**: 所有的分类结构存储在全局的 `metadata.json` 中。
    *   **物理脱钩**: 采用 UID 重命名物理文件（存放在 `.info` 目录），分类关系纯逻辑化。
    *   **并发加载**: 启动时并发扫描数万个小 JSON 文件，在内存中重建完整的关联映射。
*   **ArcMeta (SCCH 二进制驱动)**:
    *   **分布式架构**: 使用 `arcmeta_categories.scch` 存储，并支持在每个目录下生成 `.arcmeta/` 子目录存放文件级元数据。
    *   **物理对账**: 保留 `pathHint` 并配合文件系统的 FRN（File Reference Number）索引，在物理路径发生变化时具备自动修复和找回能力。
    *   **极致性能**: 所有数据在启动时由 `CategoryCacheManager` 一次性预读入内存，实现“零 I/O”渲染，消除了 UI 线程在滚动和切换时的卡顿。

## 2. 组织逻辑与层级 (Organization & Hierarchy)
*   **Eagle**:
    *   **平级模块**: 明确划分为文件夹、智能文件夹、标签组三个核心模块。
    *   **快速访问**: 在 JSON 中显式定义一个 `quickAccess` 数组来记录用户置顶的分类。
*   **ArcMeta**:
    *   **镜像模式 (Mirror Mode)**: 采用“实体项+镜像项”的组织逻辑。置顶分类会镜像显示在“快速访问”中，但在“我的分类”中依然保留其完整的父子嵌套树状结构。
    *   **原生集成**: 侧边栏集成了“物理书签”（FavoritesRepo），允许用户将原始文件路径直接固定在侧边栏，这种跨越“库”限制的特性更适合专业文件管理。

## 3. 功能特性差异 (Feature Set)
| 特性 | Eagle | ArcMeta |
| :--- | :--- | :--- |
| **智能文件夹** | 支持复杂布尔逻辑过滤 | 预设系统分类 (未分类/未标签等) |
| **加密锁定** | 预留接口，支持密码提示 | **底层 SCCH 加密**，支持 UI 锁图标展示 |
| **计数逻辑** | 统计库内所有关联项 | **纯文件计数**，严格排除文件夹干扰 |
| **视觉呈现** | 主色调定义 | **感知亮度算法**，自动计算高对比度图标颜色 |
| **稳定性** | 依赖 Node.js 异步 I/O | 引入 **UsnWatcher 节流机制**，防止 USN 突发导致 UI 崩溃 |

## 4. 结论
Eagle 追求的是**“封闭库”**管理的灵活性，其侧边栏是逻辑层面的终点；而 ArcMeta 追求的是对**“原生文件系统”**的高性能增强，其侧边栏是一个动态的、具备物理修复能力的管理中枢。ArcMeta 通过 SCCH 二进制架构在处理海量数据（百万级）时，相比 Eagle 的 JSON 模式具有更低的内存开销和更快的随机访问性能。
