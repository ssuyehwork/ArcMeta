# 用户需求记录 (User_Requirement_Record.md)

## 1. 核心需求：托管库物理与逻辑深度联动

### 1.1 场景一：同步创建与 1:1 镜像结构
- **触发点**：用户在盘符栏右键菜单选择“创建托管文件夹”，或在托管库内创建子文件夹。
- **物理行为**：在对应盘符根目录创建 `ArcMeta.Library_[盘符]`，或在其内部创建子目录。
- **期望逻辑行为**：
    - **初始同步**：创建物理根文件夹时，同步创建同名顶层分类。
    - **结构镜像**：托管库内物理子文件夹的数量、层级必须与侧边栏子分类完全一致。物理上每增加/删除一个子文件夹，逻辑分类应实时同步。
    - **递归入库**：当整个文件夹移入托管库时，系统应递归处理其内部所有子项，确保完整入库。

### 1.2 场景二：双向重命名同步
- **物理驱动逻辑**：用户在第三方（如资源管理器）修改托管库内文件夹名称，应用感知（USN 监听）并自动更新侧边栏分类名。
- **逻辑驱动物理**：用户在侧边栏修改分类名称，应用自动修改磁盘上对应的物理文件夹名称。

### 1.3 场景三：根目录强制保护
- **规则**：`ArcMeta.Library_[盘符]` 是逻辑基石。
- **行为**：若用户在外部尝试修改该根文件夹的名称，应用必须通过监听感知，并**强制将其物理名称恢复（Rename Back）**为标准名称。

### 1.4 场景四：缺失补全 (物理驱动逻辑)
- **触发点**：系统启动或实时感知。
- **行为**：发现物理文件夹存在但逻辑分类缺失时，自动补全分类。

- 确认当前代码实现状态，并完成符合“物理驱动逻辑”目标的双向同步与根目录保护功能。
