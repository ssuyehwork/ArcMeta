# FERREX 百万级数据性能优化方案 (Analysis_Modification_Plan-22.md)

本报告针对点击“全部数据”导致的 UI 卡死、假死现象进行了深度技术审计，并提出了支持百万级数据的重构方案。

---

## 1. 核心瓶颈诊断 (为什么 9721 条就卡死？)

通过对 `ContentPanel::loadPaths` 源码的静态分析，我发现了以下 **“性能杀手”**，它们在 UI 线程（主线程）中同步执行：

1.  **磁盘 I/O 滥用 (致命)**：
    在 `for (const QString& path : paths)` 循环中，系统对每个文件都调用了 `QFileInfo::exists()` 和 `info.size()`。这意味着点击“全部数据”时，主线程瞬间发起了近万次磁盘寻道请求。
2.  **慢速 API 调用 (阻塞)**：
    *   `UiHelper::getFileIcon(path)`：调用 Windows Shell API 获取图标是非常慢的同步操作。
    *   `QDir(path).isEmpty()`：检查文件夹是否为空需要扫描子目录。
3.  **内存分配压力**：
    `m_model->appendRow(row)` 在万级循环中触发了成千上万次的视图更新信号。
4.  **同步统计逻辑**：
    在加载循环中同步计算 `stats.typeCounts`、`stats.tagCounts` 等。这些本该是后台计算的任务，却在抢夺主线程资源，导致右侧筛选器面板加载时，UI 无法响应。

---

## 2. 百万级数据下的架构进化方案

若要支持百万级数据且保证丝滑流畅，必须彻底放弃现有的“全量内存加载”逻辑。

### 2.1 引入虚表模式 (Virtual View Strategy)
*   **重构方向**：将 `QStandardItemModel` 替换为继承自 `QAbstractTableModel` 的自定义虚模型。
*   **原理**：
    *   **按需加载**：视图滚动到哪一行，模型才从数据库中取出该行的元数据。
    *   **SQL 分页**：利用 `LIMIT 50 OFFSET X` 语句，每次只从 SQLite 读取极少量数据。
    *   **内存极简**：内存中不再存储上万个 `QStandardItem` 对象，仅存储当前可见的 20-30 条记录。

### 2.2 异步元数据与统计引擎
*   **后台计算**：将筛选面板所需的统计数据（各标签数量、星级分布）移至 `WorkerThread`。
*   **增量刷新**：UI 启动时先展示“全部数据”列表，右侧筛选面板则在后台计算完成后异步刷出，互不干扰。

### 2.3 图标异步懒加载 (Lazy Icon Loading)
*   **零等待加载**：列表显示时先显示通用占位图标。
*   **任务队列**：由专门的图标线程在后台逐一获取真实图标，获取成功后通过信号通知 Model 刷新对应的 Index。

## 3. 具体修改路线图

### 第一阶段：紧急泄压 (Quick Win)
1.  **禁用耗时磁盘检查**：在 `loadPaths` 中移除 `QFileInfo::exists()` 和 `QDir::isEmpty()`，改为进入该项或双击时再检查。
2.  **优化图标逻辑**：仅对可见项获取图标，非可见项直接跳过。

### 第二阶段：架构重组 (Solid Fix)
1.  **重写 `getPathsBySystemType`**：
    *   **旧**：返回 `QStringList` (百万级数据会导致内存溢出)。
    *   **新**：返回 `QSqlQuery` 句柄或支持分页的游标。
2.  **实现 `FerrexVirtualModel`**：接管 `ContentPanel` 的展示逻辑，彻底隔离 UI 信号与数据查询。

---

## 4. 结论

目前的卡顿不是因为数据量大，而是因为 **“在主线程中做了太多不该做的事”**。通过实施上述**异步化**与**虚拟化**方案，FERREX 将具备处理千万级文件记录的能力，真正对标工业级文件搜索工具。
